## 빡친 상황 (SD-3041)

PC QA 단계에서 회귀 발견. 주문상세 페이지 탭 클릭 시 두 번째 탭 클릭하면 무조건 최상단으로 점프함.

인과 구조:

| 단계 | 행위자 | 행위 |
|---|---|---|
| 과거 | 본인 | 같은 점프 동작을 캐치하고 **의식적으로 제거** |
| 의도 보존 | 본인 | **PR 디스크립션에만 적음** |
| 현재 | 다른 분 | **접근성 개선** 작업하며 포커싱 로직 손댐 |
| 결과 | — | 본인이 제거했던 동작 **부활** |

PR 디스크립션은 머지 후 코드 옆에 안 보인다. 다음 작업자가 git blame까지 따라가서 PR을 읽을 가능성? 거의 0. 특히 자기 작업이 "접근성 개선"처럼 정당해 보일 땐, "이 동작이 빠져 있다 = 누락이니 추가하자"로 자연스럽게 넘어간다. 사실은 누락이 아니라 의식적 제거였는데.

---

## 패턴 명명 — "의식적 제거의 부활"

코드를 읽을 때 **빠진 동작은 두 가지 경우**가 있다:

1. **누락** — 그냥 안 짠 것. 추가하면 개선.
2. **의식적 제거** — 일부러 안 한 것. 추가하면 회귀.

코드만 봐서는 둘이 구별 안 된다. 의도가 어딘가에 박혀 있어야 다음 작업자가 분간한다. 그 "어딘가"의 선택이 곧 회귀 방어선의 강도다.

함정: **트리거가 "좋은 일"인 작업일수록 부활시킬 위험이 크다.**

- 접근성 개선
- 성능 최적화
- 리팩토링
- 코드 컨벤션 정리

이런 작업의 작업자는 "나는 좋은 일을 하고 있다"는 인지 프레임으로 들어오기 때문에, "어 이거 빠져 있네 → 채워 넣자"가 자동 반응이 된다. 의식적 제거였을 가능성 자체를 의심하지 않는다.

## 메커니즘 비교 — 의도를 어디에 박는가

| 메커니즘 | 다음 작업자가 보는가 | 휘발 위험 | 검증 시점 |
|---|---|---|---|
| 머릿속에만 | 안 보임 | 즉시 | — |
| PR 디스크립션 | git blame 따라가야 보임 (사실상 안 봄) | 머지 직후 | — |
| 코드 주석 (라인 옆) | 같은 라인 손대면 보임 | 함수 통째 갈아엎으면 사라짐 | 사람 (PR 리뷰) |
| Characterization test | 동작 깨지면 자동 알림 | 테스트 자체 삭제 시 사라짐 | CI |
| ADR / 결정 문서 | 코드 가까이 안 두면 안 봄 | 문서 자체 휘발 | — |

**단일 메커니즘의 함정**:
- 주석만 있으면 — 리팩토링하다 함수 통째 갈아엎히면서 사라짐
- 테스트만 있으면 — "이 테스트 왜 있지?" 모르고 의도 모른 채 삭제
- ADR만 있으면 — 코드 옆에 없으면 다음 작업자 동선에 안 들어옴

→ **둘 이상 조합** 필요. 가장 강력한 조합은 **코드 주석 + characterization test**.

```ts
// 의도적으로 scrollIntoView/focus 트리거하지 않음.
// 탭 클릭 시 최상단 점프 회귀 방지 (SD-3041, 과거에도 동일 이슈 발생).
const handleTabClick = (tabId: string) => { ... };
```

```ts
it('두 번째 탭 클릭해도 최상단으로 점프하지 않는다', () => {
  // 과거 회귀 사례 SD-3041. 의도적으로 점프 동작 제거됨.
  ...
});
```

주석을 못 봐도 테스트가 잡고, 테스트가 사라져도 주석이 의도를 알린다.

## 다음에 써먹을 곳

### 의식적 제거를 결정한 순간 박을 체크리스트

- [ ] 코드 주석으로 **무엇을 / 왜** 안 했는지 (이슈 키 포함)
- [ ] Characterization test로 **그 동작이 부활하면 깨지게** 박음
- [ ] PR 디스크립션은 **보조** — 코드 주석·테스트로 박은 다음 PR에 추가 컨텍스트만

### 트리거 함정 인식

다음 작업이 "좋은 일"로 분류되는 카테고리에 들어갈 때, **그 코드 영역의 빠진 동작이 누락인지 의식적 제거인지 한 번 더 의심**:

- 접근성 개선 (포커싱·키보드 네비·스크린리더)
- 성능 최적화 (불필요해 보이는 가드·useEffect)
- 리팩토링 (안 쓰이는 것 같은 분기)
- "이 동작 빠진 거 같은데 추가할까?" 충동

### 푸시백 연결 — 회귀 사례 데이터화

회귀 사례를 누적 기록해두면 외부 요청자 푸시백의 객관 근거가 된다.

> "이 영역 최근 N건 변경 중 X건이 회귀로 이어짐 (SD-3041 등). 변경 자체 30분 + 회귀 fix 평균 Y시간."

상세는 `notes/collaboration/qa-phase-design-change-pushback.md` "회귀 비용" 섹션 참조.

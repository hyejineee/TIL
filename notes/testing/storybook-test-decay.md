## 회의감의 정체

스토리북 테스트를 쓰면서 "어차피 곧 디자인 갈아엎히면 다 버릴 텐데 왜 쓰지?"라는 의문이 들었다. 수명이 짧아 보이는 테스트에 시간을 쓰는 게 정당화되는가.

결론: **테스트의 종류를 분리하지 않고 묶어서 보면 회의감이 정당해진다.** 분리하면 회의감의 절반은 사라지고, 절반은 테스트가 풀 수 없는 문제로 남는다.

## 두 종류로 자르는 기준

> 이 테스트가 검증하는 *행동*이 디자인과 분리되어 있는가?

| 분류 | 디자인 변경 후 운명 | 예시 |
|---|---|---|
| **인터랙션 / 상태 동작** | 살아남음 | 버튼 클릭 → 모달 열림, 폼 제출 → API 호출, 토글 → 상태 변화 |
| **비주얼 스냅샷 / props 조합 가시화** | 휘발 | 색상/간격/레이아웃 스냅샷, "이 props 조합에서 이렇게 보임" 카탈로그 |

전자는 디자인이 바뀌어도 검증하려는 행동이 그대로다. 버튼 위치가 바뀌어도 "클릭 → 상태 변화" 로직은 변하지 않는다. **버튼 위치를 query하지 말고 role·label로 query하면** 디자인 의존도가 더 낮아진다.

후자는 디자인이 SoT(Source of Truth)인 것에 또 하나의 SoT를 만든 셈이라, 디자인이 움직이는 순간 동기화 부채가 된다.

## 테스트가 못 담는 것

테스트가 잘 담는 건 *what* — "이 동작이 일어나야 한다." 잘 못 담는 건 *why* — "왜 이런 UI인가, 어떤 사용자 문제를 푸는가."

테스트 이름에 "should show error toast on submit failure"라고 적어도 *왜* 토스트인지, *왜* 인라인 에러가 아닌지는 사라진다. 그건 PRD/ADR/Figma 코멘트의 영역이다. → [`notes/collaboration/prd-adr-management.md`](../collaboration/prd-adr-management.md)

## 다음에 써먹을 곳

- 스토리북 테스트를 새로 추가할 때: **"디자인이 바뀌면 이 테스트가 살아남는가?"** 한 번 자문. No면 작성 비용 대비 회수 기간이 짧다는 뜻 → 작성 안 함 또는 최소화.
- 기존 테스트를 정리할 때: 비주얼 스냅샷·props 카탈로그 위주의 것은 과감히 버려도 무방. 인터랙션 테스트는 디자인 격변에도 데려간다.
- 회의감이 다시 들면 분류부터 — 회의감은 보통 후자에서 온다.

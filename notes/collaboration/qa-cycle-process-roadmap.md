## 배경 — 매번 같은 협상이 반복된다

같은 사이클에서 정리한 4개 노트:

- `notes/collaboration/ship-before-polish.md` — 본인 폴리싱 멈추기
- `notes/collaboration/low-roi-request-pushback.md` — 일반 ROI 낮은 요청 푸시백
- `notes/collaboration/qa-phase-design-change-pushback.md` — QA 페이즈 외부 폴리싱 푸시백 + 회귀 비용
- `notes/testing/intentional-removal-regression.md` — 의식적 제거 부활 회귀 + 의도 보존

다 "**개인이 어떻게 반응할지**" 패턴이다. 이걸로 매 사이클 같은 협상을 다시 한다. 매 사이클 빡친 진짜 이유는 — **같은 협상을 다시 해야 해서**다.

→ 진짜 시간 절약은 협상을 잘하는 게 아니라 **협상 자체를 없애는 프로세스 장치**에서 나온다.

> **개인 푸시백은 매번 협상이고, 프로세스 장치는 한 번 합의로 영구 푸시백.**

---

## 진단 — 4가지 프로세스 결함

| 결함 | 증상 | 관련 노트 |
|---|---|---|
| 1. **변경 분류 게이트 없음** | 신규 UI 변경이 "버그" 티켓으로 들어옴 (예: SD-3043) | qa-phase-pushback |
| 2. **변경 임팩트 검증 없음** | 데이터 없는 폴리싱이 자동 통과 | ship-before-polish, low-roi-pushback |
| 3. **의도 보존 메커니즘 없음** | "의식적 제거"가 다음 작업자에게 안 보임 (예: SD-3041) | intentional-removal-regression |
| 4. **회귀 데이터 미누적** | 같은 영역 반복 회귀를 객관화 못함 → 푸시백 정량 근거 없음 | (이번 로그 시작) |

---

## 처방 — 5개 장치

### 본인 통제 가능 (합의 없이 지금부터 가능)

**A. PR 템플릿 — 의식적 제거 체크박스**
- 본인 PR 템플릿에 추가: "이 PR에서 의도적으로 제거·생략한 동작이 있는가? 있다면 어디(코드 주석 / 테스트)에 박았는가?"
- 비용 거의 0. 회귀 만들어서 욕먹는 것보다 체크박스 한 줄이 싸다.
- 결함 3 직접 차단.

**B. 회귀 사례 로그 누적**
- `notes/testing/intentional-removal-regression.md` "회귀 사례 로그" 섹션부터 시작 (이미 SD-3041 첫 항목).
- 5건 쌓이면 별도 파일로 분리. 영역별 회귀율이 보이기 시작.
- 결함 4 직접 차단. **모든 푸시백의 정량 근거 공급원**.

**C. 의식적 제거 시 주석 + characterization test 둘 다**
- 본인 코드부터 즉시 적용.
- 다음 작업자한테 "이거 의도예요" 신호를 코드에 박음.
- 결함 3 보강 (PR 템플릿이 챙기는 절차 + 코드 자체에 박는 흔적).

→ A·B·C는 **혼자 시작 가능**. 데이터 쌓이면 D·E 합의 자료가 된다.

### 팀 합의 필요 (데이터 쌓인 후 제안)

**D. 변경 요청 템플릿 — 디자인·기획 측**
- 변경 요청 들어올 때 다음 필드 강제:
  - **성격**: 버그 / 누락 정정 / 위치 이동·시각 강조 / 정량 임팩트
  - **사용자 임팩트 근거**: 데이터·CS·사용자 조사
  - **영향 범위**: 화면·컴포넌트
- 필드 비어 있으면 안 받음 → 객관 영역으로 자동 이동.
- 푸시백을 매번 코멘트로 다는 게 아니라 **템플릿이 알아서 푸시백**.
- 결함 1·2 동시 차단.

**E. QA 페이즈 변경 동결 룰**
- QA 부모 티켓 열리는 시점부터 신규 UI 변경은 **다음 사이클로 디폴트 이월**.
- 예외만 임팩트 근거 + PM 승인으로 통과.
- 명문화 후엔 매번 협상 안 함.
- 결함 1·2 영구 차단.

## 정착 로드맵 (`prd-adr-rollout` 원리 적용)

본인이 이미 정리한 "한 번에 다 깔면 정착 실패. 작게 시작 → 모멘텀 → 확장":

| 시점 | 깔 장치 | 비용 | 산출물 |
|---|---|---|---|
| **이번 주** | A (PR 템플릿) + C (주석+테스트) | 거의 0 | 회귀 데이터 발생 시작점 |
| **2~3주** | B 누적 (3~5건) | 회귀 발생 시 1줄 추가 | 영역별 패턴 가시화 |
| **1~2달 (회고/1on1 시점)** | D 제안 | 합의 미팅 1회 | 변경 요청 템플릿 정착 |
| **D 정착 후** | E 동결 룰 합의 | 합의 미팅 1회 | 룰 명문화 |

E를 1단계에 던지면 "개발팀이 일 안 받겠다는 거냐"로 받아치기 쉽지만, B 데이터로 시작하면 **객관적 비용을 본 다음의 자연스러운 결론**이 된다.

## 다음에 써먹을 곳

### 협상 → 장치 전환 체크리스트

같은 종류의 푸시백을 **두 번 이상** 했으면 시그널:

- [ ] 같은 패턴의 협상을 또 하고 있나? → 개인 패턴이 아니라 **프로세스 결함** 신호
- [ ] 그 결함을 막을 장치가 있나? (템플릿 / 룰 / 데이터 누적)
- [ ] 본인 통제 가능한 장치부터 깔고 있나? → 합의 안 받고 시작
- [ ] 데이터 누적이 진행 중인가? → 합의 자료 공급
- [ ] 합의 시점·자리(회고/1on1)는 정해졌나? → 즉흥 제안 X

### 합의 미팅에서 쓸 한 줄

> "최근 N건 회귀 중 X건이 [영역] 폴리싱발이고, 변경당 평균 Y시간 회귀 fix 들어갔습니다. [장치] 도입하면 이 시간을 줄일 수 있을 것 같습니다."

추상 제안 ❌ → 데이터 + 구체 장치 ✅. `low-roi-request-pushback`의 객관 공넘기기 원리 그대로, 다만 **이번엔 본인이 거는 쪽**.

## 메타 — 노트의 진화 단계

본인 협업 노트들의 발전 궤적:

```
1단계 (개인 행동 패턴)
  ├─ ship-before-polish      ← 자기 자신 멈춤
  ├─ low-roi-request-pushback ← 일반 푸시백 패턴
  └─ qa-phase-pushback        ← 특정 페이즈 푸시백

2단계 (회귀·의도 보존)
  └─ intentional-removal-regression ← 코드 차원 방어

3단계 (프로세스 장치) ← 이 노트
  └─ qa-cycle-process-roadmap  ← 1·2단계를 시스템화
```

각 단계는 다음 단계를 위한 데이터·합의 근거를 만든다. 1단계 없으면 2단계에 패턴을 못 보고, 2단계 없으면 3단계에 정량 근거가 없다.

다음(4단계)이 있다면 — **회사 차원 정책**이거나, 다른 팀에 같은 로드맵을 이식하는 단계. 거기까지 가려면 3단계가 6개월 이상 운영되며 효과 데이터가 나와야 함.

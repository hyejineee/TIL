## 문제: *why*는 어디로 사라지는가

스토리북 테스트가 *why*를 못 담는다는 걸 인정하는 순간 다음 질문이 따라온다 — **"그럼 *why*는 어디에 있어야 하나?"** 우리 조직은 PRD·ADR·Figma 코멘트가 노션·지라·슬랙·PR 본문에 흩어져 있고, 배포 직후부터 사실상 휘발된다. 어찌어찌 Figma 파일을 찾을 수는 있는데 시간이 너무 오래 걸린다.

산재의 4가지 패턴:

1. **도구 분산** — 같은 결정의 조각이 노션/Figma/슬랙/지라/PR description에 나뉘어 존재
2. **시간 휘발** — 배포 후 "끝난 일"로 분류돼 아카이브 깊숙이 묻힘
3. **코드 단절** — 코드만 보면 *why*에 도달할 진입점이 없음
4. **검색 진입점 부재** — "어디에 있더라"부터 찾아야 함

## 장치 1: 단일 위치 < 단일 인덱스

도구를 한 곳으로 통합하는 건 비현실적이다 (디자인은 Figma, 백로그는 지라, 협업은 슬랙). 대신 **"어디에 있는가"를 한 곳에 인덱싱**한다. 이 TIL 레포의 `INDEX.md`처럼 — 본문은 노션/Figma에 두되 인덱스는 한 군데. 인덱스가 없는 도구 통합은 통합이 아니라 새로운 산재다.

## 장치 2: 코드 → 문서 역방향 링크

컴포넌트/기능에서 출발해 *why*까지 한 번에 도달 가능해야 한다. 컴포넌트 README나 파일 헤더 주석에:

```ts
/**
 * @prd https://...
 * @figma https://figma.com/...?node-id=...
 * @adr docs/adr/0042-toast-vs-inline-error.md
 */
```

또는 Figma Code Connect로 컴포넌트 ↔ Figma 노드 ID를 직접 매핑하면 IDE에서 바로 디자인으로 점프할 수 있다. 피그마 검색 시간이 사라지는 가장 큰 레버다.

## 장치 3: ADR을 코드 가까이

PRD는 노션/지라에 두더라도, **결정 기록(ADR)은 리포 안에** 둔다. `docs/adr/0001-xxx.md` 같은 Markdown ADR 컨벤션. 장점:

- 코드와 함께 버전 관리됨 → "이 결정이 언제 바뀌었나" git blame 가능
- PR diff에 결정이 같이 올라감 → 리뷰어가 코드와 의도를 동시에 봄
- 검색이 `grep`/IDE로 끝남 → 도구 사이 점프 비용 0

ADR은 짧게: 컨텍스트 / 결정 / 대안 / 결과 4문단이면 충분하다. 길어지면 안 쓴다.

## 장치 4: PR 템플릿으로 강제

> 결정 시점에 잡지 못하면 영원히 잡지 못한다.

PR 템플릿에 "관련 PRD/ADR/Figma 링크" 필수 필드를 두고, 비어 있으면 머지 못 하게 한다. *나중에 정리* 같은 건 일어나지 않는다. 작성하는 사람의 머릿속에 가장 많은 정보가 있을 때 (= PR 올릴 때) 강제로 한 줄 받아내는 게 핵심.

## 다음에 써먹을 곳

- 신규 기능 시작 시 가장 먼저: "이 결정의 ADR은 어디에 두지?" 정한다. 위치를 정하지 않은 결정은 사라진다.
- PR 템플릿 개선 제안 시: "관련 결정 링크" 필드 추가 + 빈 머지 차단을 한 세트로.
- Figma 검색이 또 오래 걸리면: 그 컴포넌트의 README에 링크가 박혀 있어야 한다는 신호. 한 번 찾았을 때 박아두면 두 번 찾을 일은 없다.
- 관련: [`notes/testing/storybook-test-decay.md`](../testing/storybook-test-decay.md) — *why*가 테스트로 안 남는 이유.

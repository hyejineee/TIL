## 빡친 상황 (쉬이바)

`PlatformTabFilter.tsx` — 상품 목록의 모바일 플랫폼 필터 탭. 현재 로직은:

```ts
const MOBILE_DROPDOWN_THRESHOLD = 4;
const hasDropdown = platforms.length > MOBILE_DROPDOWN_THRESHOLD;
```

**플랫폼 개수**가 4개를 초과하면 드롭다운 버튼/오버레이 노출. 단순하고 잘 동작함.

기획자가 요청한 변경: "플랫폼 **너비**(overflow 여부)에 따라 드롭다운 아이콘 보였다 안보였다 해달라."

→ count 기준 → 실제 overflow 측정 기준으로 바꿔달라는 것. 빡친 이유:
- overflow 안 되는데 드롭다운이 떠 있어도 → 누르면 그냥 2열 그리드로 보임. **기능 깨짐 X**, 시각적 잉여뿐.
- overflow 되는데 드롭다운이 안 뜨면 → 사용자가 가로 스크롤로 다 봐야 함. **이쪽이 진짜 손해**.
- 즉 보수적 디폴트는 "항상 띄워두기". **"안 뜨는 것"의 손해 > "떠 있는 것"의 손해.**

그런데 굳이 ResizeObserver + `scrollWidth/clientWidth` 측정 + SSR 깜빡임 + 폰트 로딩 타이밍 처리까지 들고 와서, 시각적 잉여 1개 제거하자고? 자원 낭비 맞다.

---

## 기술적으로 다시 본 ROI

### 비용 (한 번 짜면 끝이 아님)
- ResizeObserver 1개 + 상태 1개
- SSR 초기 hydration 시점 깜빡임 처리 (초기엔 보수적으로 띄워둘지 결정해야 함)
- 폰트 로딩 타이밍에 따라 측정값 흔들림 → `document.fonts.ready` 대기
- **유지보수 부채**: 다음에 누가 손댈 때마다 "왜 안 뜨지/왜 뜨지" 디버깅 거리

### 이득
- overflow 안 될 때 아이콘 1개 안 보이게 함

### 결론
**이득 < 비용.** "있어도 그만"인 케이스에 측정 로직 박는 건 ROI 마이너스.

> 일반론으로는 "JS로 너비 재서 토글" = 안티패턴이라고 답했다가, 막상 코드 보니 무조건 안티패턴은 아니었다. count threshold도 정확한 시그널은 아니라 (이름 길이 따라 흔들림) UX 정확성 측면에선 측정이 더 맞다. **다만 이 컴포넌트의 실제 사용 맥락에서 ROI가 낮은 것.** 일반론과 케이스 분석을 분리해서 봐야 한다.

## 배운 것 — 푸시백 패턴

### ❌ 안 통하는 거절
> "이거 비효율이라 안 할게요"
> "ResizeObserver 써야 하는데 그거 비싸요"

→ 주관 싸움 됨. 기획자도 "그래도 해주세요" 카드 있음.

### ✅ 통하는 거절 — 공 넘기기
> "이 변경은 overflow 안 되는 케이스에서 아이콘 1개 안 보이게 하는 게 목적인데, 그 케이스가 실사용에서 얼마나 자주 발생하나요? 자주 안 일어나면 측정 로직 추가 비용 대비 이득이 작아서, 현재 threshold 유지하거나 값을 조정하는 쪽으로 가는 게 좋아 보여요."

핵심:
1. **요청의 목적을 한 줄로 요약** (= "내가 정확히 이해했다"는 시그널)
2. **그 목적이 발생하는 빈도/사례를 데이터로 요청** (= 객관 영역으로 끌고 옴)
3. **저비용 대안 제시** (= "거절"이 아니라 "다른 길 있다")

기획자도 보통 "음, 그럼 일단 두자"가 됨. **"안 하겠다"는 주관 싸움이지만, "ROI 근거 보여달라"는 객관 싸움.** 통과율 압도적으로 높음.

---

## 다음에 써먹을 곳

- "있어도 그만 없어도 그만" 류 요청 전반 — 마이크로 인터랙션, 시각 디테일, 엣지 케이스 처리
- 요청의 **사용자 가치**가 명시 안 된 채로 들어온 변경 ("이렇게 해주세요" 만)
- 기존 단순한 휴리스틱(threshold, count, 고정값)을 **정확한 측정값으로 바꿔달라**는 요청 — 거의 항상 ROI 점검 대상
- 보수적 디폴트가 이미 "안전한 잉여"인 케이스 — 잉여를 제거하는 비용 vs 잉여의 손해 저울질

---

## 메타 교훈

코드 안 읽고 일반론으로 답하면 틀린다. 처음엔 "JS로 너비 재기 = 안티패턴"이라고 받아쳤는데, 코드 보니 현재 count threshold도 부정확한 휴리스틱이었다. **일반론은 출발점이고, 실제 푸시백은 코드+사용 맥락 본 다음에 해야 정확하다.** 빡친 감정이 맞아도 근거는 따로 챙겨야 함.

## 빡친 상황

PC QA 부모 티켓(`SD-2500`) 아래에, **이슈 타입은 "버그"인데 내용은 신규 UI 변경**인 하위 티켓(`SD-3043`)이 떨어졌다. 충전하기 화면의 금액 영역을 다시 만져달라는 요청. 변경 사유:

1. 대행사 상시 충전 혜택 안내를 충전 결제 금액 영역에 넣고, 클릭 시 상세 확인 UI로 변경
2. 충전 유도가 더 잘 보이도록 받을 수 있는 포인트 혜택과 구분

근데 1번은 **이미 충전 폼 금액 아래에 표시되고 있던 내용을 다른 위치로 옮겨달라**는 거고, 2번은 **이미 충전 유도 UI가 동작하고 있는데 시각적으로 더 강조해달라**는 거다. 즉 둘 다 정보 누락 정정이 아니라 **시각적 폴리싱**.

QA 막바지에 잔여 작업·실제 QA 케이스 검증을 끝내야 하는 시점에, 이미 동작하는 화면을 또 만지라는 요청이 "버그"라는 이름표를 달고 들어왔다.

---

## 배운 것 — 3단계 분류

UI 변경 요청을 받았을 때 즉답·즉시 작업 전에 30초만 멈추고 분류한다.

### 1. 이슈 분류 검증

티켓 타입이 변경의 실제 성격과 맞는지부터 본다.

| 티켓 표기 | 실제 내용 | 정합성 |
|---|---|---|
| 버그 | 동작 오류 / 기획 스펙 미구현 | ✅ |
| 버그 | 신규 UI 변경 / 디자인 폴리싱 | ❌ — 분류 정정 필요 |
| 개선 / 변경 요청 | 이미 동작하는 UI 손보기 | 분류는 맞음, ROI 검증 필요 |

분류가 틀리면 추적·평가 자료가 오염된다. "QA 단계에서 버그 N건 발견" 같은 메트릭이 신규 변경 요청까지 포함하게 됨. 회고·OKR·평가 근거로서 가치가 떨어진다.

→ **"이건 버그가 아니라 디자인 변경 요청 같은데, 이슈 타입 다시 잡아주실 수 있나요?"** 부터 시작.

### 2. 변경 사유 분리

요청 안에 보통 2~3가지가 섞여 있다. 분리해서 각각 분류한다.

| 변경 성격 | 처리 방향 |
|---|---|
| **정보 누락 정정** — 사용자가 못 보던 게 있었음 | ship-before-polish 예외, QA 내 처리 |
| **위치 이동 / 시각 강조** — 정보는 이미 있는데 더 잘 보이게 | 폴리싱, 배포 후 데이터 보고 판단 |
| **정량 임팩트 변경** — 전환율 등 비즈니스 지표 개선 목적 | 임팩트 근거 데이터 요구 |

함정: "충전 유도 강조"처럼 **목적은 비즈니스(전환율)인데 근거는 주관**인 케이스. "유도가 더 잘 보였으면 좋겠다"는 의견과 "유도 강화로 전환율 +X% 기대"는 다른 종류의 주장이다. 후자라면 baseline 전환율과 가설을 같이 받아야 한다.

### 3. 폴리싱이면 배포 후로 — 푸시백 코멘트

분류 결과가 폴리싱이면 거절이 아니라 **분리·연기 제안** 톤으로:

> 두 변경이 섞여 있어 분리해서 보고 싶습니다.
>
> 1. [변경 A] — 현재 UI에 이미 동일 정보가 [위치 X]에 표시되고 있어, **정보 누락 정정이 아닌 위치 이동**으로 이해됩니다.
> 2. [변경 B] — 기존 [기능]이 이미 동작 중이라, **시각적 강조 변경**에 해당합니다.
>
> 이 경우 PC QA 잔여 작업·배포 일정에 영향이 갑니다. 배포 후 실사용 데이터(전환율·이탈 지점·CS 사례) 보고 다시 판단하는 쪽으로 제안드려요. 지금 꼭 들어가야 하는 사용자 임팩트 근거가 있다면 알려주시면 재검토하겠습니다.

핵심 4가지:
1. **요청 내용을 한 줄로 요약** — "정확히 이해했다"는 시그널
2. **현재 상태와 요청 간 차이를 명시** — "이미 이렇게 동작 중"이라는 객관 사실
3. **폴리싱이라는 분류를 명시** — 거절이 아니라 카테고리화
4. **임팩트 근거 있으면 재검토** — 공을 객관 영역으로 넘김

## 회귀 비용 — 폴리싱의 진짜 가격

폴리싱 한 건의 비용은 **작업 시간만이 아니다**. 다음을 포함한다:

- 작업 시간 (보이는 비용)
- **회귀 위험** (보이지 않는 비용)
- **회귀 fix 시간** (회귀 발생 시 추가)
- **회귀 검증 시간** (변경 후 같은 코드 영역 쓰는 다른 화면 QA)

같은 사이클의 다른 폴리싱 변경(예: 접근성 개선)이 **이전에 의식적으로 제거했던 동작을 부활**시킨 회귀 사례 — `SD-3041`. 즉 폴리싱은 자기 자신만 부르는 게 아니라 **다른 폴리싱과 충돌해서 회귀를 만들어낸다**.

→ 푸시백 코멘트에 회귀 사례 데이터를 객관 근거로 사용:

> 이 변경 자체는 작업 시간 30분이지만, 같은 코드 영역에서 최근 N건 변경 중 X건이 회귀로 이어졌습니다 (예: SD-3041). 변경 시간 + 회귀 검증 시간 + 회귀 fix 시간을 합치면 ROI가 마이너스에 가깝습니다. 배포 후 실데이터 + 회귀 위험 낮은 시점에 일괄 처리 제안드려요.

회귀 사례를 누적 기록해두는 패턴은 `notes/testing/intentional-removal-regression.md`의 "회귀 사례 로그" 참조.

## 왜 중요한가

푸시백을 한 번도 안 하면 디자인·기획팀에서 **"요청 = 자동 통과"**가 학습된다. 그러면:

- 매 QA 페이즈마다 같은 패턴의 폴리싱 요구가 또 들어옴
- 개발팀은 진짜 중요한 QA·잔여 기능 검증을 못 끝내고 보여지는 거 만지는 데 시간을 씀
- 검증 없는 주관 변경이 누적되어, 배포 시점에 "이게 진짜 개선이었는지"조차 확인 불가
- 분류 오류 누적으로 메트릭(버그 N건, QA 사이클 길이 등) 신뢰도 하락

한 번이라도 **"이 변경의 임팩트 근거 보여달라"**는 객관 푸시백이 들어가면, 다음 요청 때 본인들이 미리 근거를 챙겨오기 시작한다. 이게 프로세스 개선의 시작점.

이건 `low-roi-request-pushback.md`의 원리(주관 거절 ❌ / 객관 공넘기기 ✅)와 `ship-before-polish.md`의 원리(폴리싱은 배포 후로)를 **외부 요청자 + QA 페이즈** 케이스에 결합한 것.

## 다음에 써먹을 곳

QA 페이즈 진입 후 들어오는 모든 UI 변경 요청에 30초 체크리스트:

- [ ] 이슈 타입이 변경 성격에 맞나? (버그 vs 신규 변경 vs 기획 누락) → 안 맞으면 분류 정정 요청부터
- [ ] 변경 사유가 "정보 누락 정정"인가, "위치 이동 / 시각 강조"인가? → 후자면 폴리싱
- [ ] 폴리싱이면 임팩트 근거(전환율·이탈·CS 사례)가 있나? → 없으면 배포 후로 미룸 제안
- [ ] 푸시백 코멘트는 거절이 아니라 분리·연기 제안 톤인가? → 받아주는 입장에서 받기 쉬운 형태인가

특히 **이미 자기가 작업한 화면을 다시 만지라는 요청**이 들어왔을 때가 알람 신호. "왜 또?"가 떠오르면 그 감정을 위 분류 프레임으로 변환해서 코멘트로 옮긴다.

설령 결정권자가 "그래도 둘 다 해주세요"라 답하면 그건 받되, **푸시백 기록 자체가 자기 방어 자료**가 된다. 회고·1on1·평가 때 "이 시점에 ROI 의문 제기했고, 결정권자 판단으로 진행함"이 추적된다.

## 메타 — 본인 폴리싱 vs 남의 폴리싱

- 본인 머릿속에서 올라온 "이게 더 좋겠는데?" → `ship-before-polish.md` (스스로 멈춤)
- 남이 보낸 "이렇게 바꿔주세요" → 이 노트 (분류 후 푸시백)
- 둘 다 같은 원리: **데이터 없는 UX 변경은 개선인지조차 모르는 주관 변경**. 다만 자기 자신은 그냥 멈추면 되지만, 남에게는 분류·연기 제안을 통해 객관 영역으로 옮겨야 한다.

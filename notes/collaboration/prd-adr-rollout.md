## 한 번에 다 깔면 안 되는 이유

[`prd-adr-management.md`](./prd-adr-management.md)의 4가지 장치(단일 인덱스 / 역방향 링크 / 코드 가까이 ADR / PR 템플릿 강제)를 한 번에 도입하면 거의 100% 정착 실패한다. 이유는 비용이 앞에 몰리고 효용이 뒤에 오기 때문 — 1주차에 합의·인덱스 구조·역방향 링크를 다 잡으려다 첫 ADR을 못 쓰고 끝난다.

**저비용·고ROI 순서로 한 장치씩 깐다.** 한 레포(예: modoogram-frontend)에서 파일럿 검증 후 확장.

## 도입 순서

| 순서 | 장치 | 비용 | 합의 필요? | 시점 |
|---|---|---|---|---|
| **1** | PR 템플릿 강제 | 매우 낮음 | 거의 X | 이번 주 |
| **2** | ADR 디렉토리 + 메타 ADR | 낮음 | 가벼움 | 이번 주 |
| **3** | 코드 → 문서 역방향 링크 | 점진 누적 | 자연 발생 | 1번이 강제하면 따라옴 |
| **4** | 단일 인덱스 | 합의 비용 큼 | 필요 | ADR 5+개 쌓인 뒤 |

핵심 통찰: **PR 템플릿 1개가 나머지 3개의 강제 장치 역할을 한다.** 머지 전 "관련 결정 링크" 빈 칸을 못 비우면 작성자가 알아서 ADR 만들고 코드에 링크 박는다. 위에서 강제하지 않아도 자생적으로 누적됨.

## 1주차 구체 액션 (단 3개 파일)

### (A) `.github/pull_request_template.md` 섹션 추가

```markdown
## 관련 결정 / 디자인
<!-- 비어 있으면 "해당 없음" 명시. 빈칸 머지 금지. -->
- PRD/이슈:
- Figma:
- ADR:
```

작성자 본인의 *나중에 찾는 시간*을 줄이는 거라 반발 거의 없음.

### (B) `docs/adr/0000-record-architecture-decisions.md`

"왜 ADR을 도입하는가"를 ADR 자체로 남긴다. 이게 첫 본보기 톤이 되므로 **4문단(컨텍스트 / 결정 / 대안 / 결과)·30줄 이내**로 짧게. 첫 ADR이 길면 다음 ADR이 안 나온다.

### (C) `docs/adr/_template.md`

4섹션 빈 템플릿. 복사 붙여넣기로 새 ADR 시작 가능.

이 3개로 1주차 끝. 팀 공유 1번 + "다음 PR부터 적용" 공지.

## 2~4주차: 첫 실전 ADR

진짜 결정이 일어나는 다음 PR (예: 토스트 vs 인라인 에러 / 라우팅 라이브러리 선택)에서 ADR 1개 시범 작성. 동시에 코드 헤더에 `@adr docs/adr/0001-xxx.md` 박아서 역방향 링크도 시범. **한 번 이 패턴이 PR에 보이면 따라하는 사람 생긴다** — 위에서 강제하지 않아도.

## 흔한 함정 3가지

1. **무거운 템플릿** — ADR을 *문서*로 보면 길어진다. ADR은 *기록*이다. 4섹션 넘지 말 것.
2. **모든 컴포넌트에 한꺼번에 역방향 링크 박기** — 절대 X. 새로 만지는 것부터 누적해야 부채 안 생김.
3. **인덱스 먼저 만들기** — 인덱싱할 ADR이 5개도 없는데 디렉토리 구조부터 고민하다 시작이 안 됨. ADR 5+개 쌓인 뒤 README에서 인덱싱.

## 다음에 써먹을 곳

- 새 레포에 *why* 시스템 도입할 때: 이 순서표 그대로. 한 번에 다 깔지 말 것.
- ADR 도입이 흐지부지될 때: 보통 (1)템플릿이 무겁거나 (2)PR 템플릿 강제가 없어서임. 이 두 가지부터 점검.
- 관련: [`notes/collaboration/prd-adr-management.md`](./prd-adr-management.md) — 원리(4가지 장치).
